## Angular Signals: Как мы переписали приложение и почему за этим будущее

Я пришёл в текущую компанию около полутора лет назад, как раз в тот момент, когда Angular Signals только начинали набирать обороты. Надо отдать должное создателям фреймворка SolidJS - именно они первыми популяризировали эту концепцию, а команда Angular элегантно подхватила эстафету.

Сама идея сигналов куда старше, реактивные наблюдаемые значения были еще в Knockout в 2010 году, а само слово signal закрепилось в библиотеке S.js. В своем RFC по сигналам команда Angular написала, что вдохновлялась Preact, Vue и Solid.js.

И это в корне изменило то, как работает наша аппликация и насколько проще стала разработка. Мы начали внедрять их постепенно: сначала писали на сигналах только новые фичи, затем плавно перешли к рефакторингу старого кода. Сегодня мы почти полностью переехали на сигналы.

## RxJS vs Signals: В чем реальная разница?

Раньше мы повсеместно использовали связку Observable и RxJS. Разницу между старым подходом и сигналами можно сравнить с водопроводом и таблицей Excel:

**RxJS (Потоки событий):** Это сложная система труб. Данные текут по ним, и вам нужно вовремя подписаться, чтоб поймать поток, попутно следя за тем, чтобы ничего не «прорвало» (утечки памяти).

**Signals (Состояние):** Это умная ячейка в Excel. Вы задаете формулу, и если данные в соседних ячейках меняются, ваша ячейка обновляется сама. Значение доступно вам всегда, здесь и сейчас.

RxJS, по правде говоря, дал нам эту реактивность еще давно, но ценой немалой церемонии: подписки, отписки, десятки операторов. Сигналы дают ту же реактивность, только в куда более простой и синхронной обертке: сигнал - это всегда актуальные значения, которые можно просто прочитать, без подписок и ручного удаления.

## Наш путь миграции

В связи с этим мы полностью пересмотрели архитектуру. Вместо того чтобы строить сложные асинхронные цепочки ожиданий и блокировок, мы просто запрашиваем данные, показываем лоадер и спокойно пьем кофе, пока интерфейс сам не отреагирует на изменение стейта. Кстати, для этого в Angular теперь есть resource() и httpResource() - они отдают состояние загрузки, ошибку и сами данные прямо в виде сигнала.

## А как же старый добрый RxJS?

Означает ли наш переход на сигналы, что RxJS пора полностью списать со счетов? Абсолютно нет! RxJS - это мощнейший и критически важный механизм, который остается незаменимым для решения специфических и сложных задач: обработки высокочастотных событийных потоков, WebSocket-соединений, race conditions, debounce-фильтрации или сложной координации параллельных асинхронных запросов.

В программировании не бывает так, чтобы мы всегда шли только по одной дороге. Иногда дороги нужно просто ровно заасфальтировать (и здесь простые и понятные сигналы идеальны для синхронизации состояния), а иногда - строить сложные подвесные мосты из RxJS-операторов для гибкого управления потоками данных. Эти инструменты созданы не для конкуренции, а для работы в связке. Но их архитектурный симбиоз - это уже отдельная история.

## Разрушаем старые табу: Функции в темплейтах

Помните один из самых частых вопросов на интервью? «Почему нельзя вызывать функции в HTML-шаблоне Angular?» Раньше это было строгим табу: каждое срабатывание Change Detection вызывало функцию заново, создавая огромную нагрузку. Нам приходилось плодить лишние переменные в компоненте и биндить их в шаблон, раздувая код.

С сигналами этой проблемы больше нет. Вызов mySignal() в шаблоне - это абсолютно безопасно, так как это просто возврат уже хранящегося значения: O(1), a computed вдобавок кеширует (мемоизирует) результат и пересчитывает его только тогда, когда реально изменилась одна из зависимостей. Поэтому читать сигнал из шаблона дешево, и все работает быстро.

## Что происходит под капотом

**Glitch-Free выполнение (Push/Pull гибрид):** Сигналы Angular работают по смешанной модели. Когда сигнал меняется, он не проталкивает новые вычисления сразу. Вместо этого он лишь помечает зависимые сигналы как «грязные» (Dirty). Перерасчет происходит только тогда, когда значение реально запрашивается (Pull). Это исключает моргание интерфейса из-за промежуточных состояний.

**Динамический граф зависимостей:** Сигналы перестраивают свои подписки при каждом вычислении. Если в computed есть условие if (condition()) { return data() }, то при condition() === false сигнал отпишется от data(). Никакой работы вхолостую.

## Прощай, Default Change Detection

Несмотря на всю магию сигналов, мой вам совет: перестаньте использовать стандартный механизм обнаружения изменений.

С выходом Angular v22 команда Angular официально отразила истинную суть ChangeDetectionStrategy.Default, переименовав его в Eager («жадный»), а старое название пометила как deprecated. И это идеальное описание - он жадно проверяет все дерево компонентов при любом чихе. К счастью, **`OnPush` официально стал стратегией обнаружения изменений по умолчанию для новых проектов**, а значит, новые проекты получают эти преимущества производительности прямо из коробки.

Сигналы же обеспечивают Fine-grained reactivity (точечное обновление только изменившегося куска DOM). Используйте OnPush и двигайтесь в сторону Zoneless-приложений, оставив Eager проверки в прошлом. Это открывает путь к уверенному переходу на **Zoneless**-приложения. Переход на Zoneless означает отказ от библиотеки `zone.js`, которая исторически перехватывала каждое асинхронное событие (Monkey patching): клики, таймеры, HTTP-запросы, чтобы принудительно запускать глобальные проверки. Удаление `zone.js` также уменьшает размер бандла (примерно на ~30 кБ raw). Сигналы явно сообщают Angular, где именно и когда изменилось состояние, обеспечивая гораздо более легкую и быструю среду выполнения без глобальных накладных расходов.

## Экосистема и пара советов

Мы получили мощнейшие инструменты вроде NgRx SignalStore и Signal Forms (которые точно заслуживают отдельного поста). А пока вы пишете код на сигналах, помните о двух вещах:

**Используйте untracked:** Когда работаете с effect(), легко случайно подписаться на сигнал, который не должен вызывать повторный запуск эффекта. untracked() позволяет игнорировать такие чтения и снимает лишнюю нагрузку.

> *Пример:* Если вам нужно логировать вызов API всякий раз, когда меняется `userName()`, но также необходимо прочитать сигнал `sessionId()` *без* того, чтобы изменение сессии повторно триггерило этот лог, оберните его в `untracked(() => sessionId())`.

```typescript
effect(() => {
  const name = this.userName();
  untracked(() => {
    console.log(`User name changed to ${name}, session ID: ${sessionId()}`);
  });
});
```
Изучите linkedSignal: Эта фича делает магию, позволяя создавать сигналы на основе текущих, но с возможностью их ручного обновления.  

Пример: Если у вас есть сигнал shippingOptions, который должен автоматически сбрасываться в значение по умолчанию всякий раз, когда меняется shippingOptions(), но при этом позволяет пользователю вручную выбирать и переопределять выбранную опцию, linkedSignal безупречно справляется с этой синхронизацией состояния без вложенных ручных эффектов.


```typescript
import { Component, signal, linkedSignal } from '@angular/core';
@Component({
    selector: 'app-shipping-picker',
    template: `
 <p>Selected: {{ selectedOption() }}</p> 
 @for (option of shippingOptions(); track option) 
 {
 <button (click)="changeShipping(option)"> {{ option }} </button>
 } 
 <button (click)="loadExpressOptions()">
 Load Express Options
 </button> `,
})
export class ShippingPickerComponent {
    shippingOptions = signal(['Ground', 'Air', 'Sea']);

     // Сбрасывается на первую опцию при изменении shippingOptions, 
     // но пользователь может вручную выбрать любую опцию 
     selectedOption = linkedSignal(() => this.shippingOptions()[0]); 

     // Пользователь вручную выбирает опцию из компонента 
     changeShipping(option: string) { 
     this.selectedOption.set(option); 
     // ручное переопределение (manual override) 
     } 

     // Симулирует загрузку новых опций из API 
     loadExpressOptions() { 
     this.shippingOptions.set([
     'Carrier Pigeon (Weather Permitting)', 
     'Trained Falcon (Slightly Aggressive)', 
     'Yeeted via Catapult (For robust packages only. Parachute does not always open, backup available for an extra fee)', 
     'Delivered by Sonic the Hedgehog (If he takes damage, your package will scatter everywhere)']); 
     // selectedOption автоматически сбрасывается на первую опцию
      }
 }
 ```
